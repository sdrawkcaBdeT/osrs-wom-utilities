# Ecosystem Data Pipeline Architecture (Track B)

SCOPE LIMITATION: This document ONLY covers the Track B (Economy) pipeline originating from the gp-per-hour plugin. For combat telemetry schemas, refer to ECOSYSTEM_TELEMETRY_PIPELINE.md.

This document maps out the current RuneLite data collection ecosystem, focusing on how the unmodified gp-per-hour (GPPH) plugin session data is ingested, expected schemas, and their downstream destinations. **Any modifications to RuneLite data collection must maintain backward compatibility with these structures.**

---

## 1. Raw Ingestion Layer
**Entry Script:** `get_gpph.py`
**Source:** RuneLite `.properties` profiles (e.g., `C:\Users\Teddy\.runelite\profiles2\$rsprofile--1.properties`).

### Extraction Logic
The script scans the file for lines starting with the specific RuneLite plugin prefix:
`gpperhour.rsprofile.t-qiiBcR.session_stats=`

The value is expected to be a valid JSON object. The parser extracts the following keys from this JSON:
*   `sessionID`: Unique identifier (UUID).
*   `sessionName`: Human-readable name.
*   `sessionSaveTime`: End timestamp (in ms).
*   `sessionRuntime`: Duration of the session (in ms).
*   `tripCount`: Number of banking trips.
*   `netTotal`: Overall net profit.
*   `totalGain`: Gross profit.
*   `totalLoss`: Total cost of supplies.
*   `initialQtys`: Dictionary of `{ "item_id": quantity }` indicating starting gear/supplies.
*   `qtys`: Dictionary of `{ "item_id": quantity }` indicating ending gear/loot.

### Destination (Output)
`get_gpph.py` processes this raw JSON into two distinct relational CSVs:
1.  **`gpph_sessions.csv`**: Contains top-level session metadata.
    *   *Schema:* `session_uuid`, `name`, `local_start_time`, `wiki_pricing_timestamp`, `duration_seconds`, `trip_count`, `net_profit`, `total_gain`, `total_loss`
2.  **`gpph_ledger_raw.csv`**: Contains delta calculations for every individual item.
    *   *Schema:* `session_uuid`, `item_id`, `qty_delta`, `category` (LOOT or SUPPLY)

---

## 2. Enrichment Layer
**Entry Script:** `enrich_gpph.py`
**Source:** `gpph_sessions.csv`, `gpph_ledger_raw.csv`, `items.csv`, and `price_snapshots/*.csv`

### Processing Logic
This script acts as the financial engine. It merges the literal item ID deltas with historical point-in-time pricing data.
*   Converts the 5-minute `wiki_pricing_timestamp` bucket into an hourly `hourly_ts` bucket.
*   Looks up the specific `item_id` in the corresponding historical price snapshot (from `price_snapshots/prices_{ts}.csv`).
*   Matches IDs to names using `items.csv`.

### Destination (Output)
It generates a unified, priced ledger:
1.  **`gpph_enriched.csv`**:
    *   *Schema:* Includes all ledger columns + `session_name`, `local_start_time`, `wiki_pricing_timestamp`, `item_name`, `hourly_ts`, `hist_price_unit`, `total_value`

---

## 3. Normalization & Machine Learning Layer
**Entry Script:** `normalize_sessions.py`
**Source:** `bbd_data/*.json` (Battle-Bot Data telemetry) AND `gpph_enriched.csv`

### Processing Logic
This script binds the GPPH supply costs to the high-frequency telemetry data generated by our custom plugins.
*   Loads telemetry JSONs expecting keys: `start_time`, `end_time`, `total_kills`, `active_seconds`, `event_timeline` (for bank time), `config` (for One-Hot Encoding gear setups), `loot_summary`, `total_attacks`, `theoretical_stats`.
*   Scans `gpph_enriched.csv` for supply costs (`qty_delta < 0`) occurring within a +/- 2-minute window of the telemetry session's `start_time` to calculate precise `actual_supply_cost`.
*   Calculates advanced metrics: Theoretical GP/hr (T-NGP/hr), RNG variance, Active Time to Bank (ASTB), and Missed Attacks per hour (`miss_per_hr`).

### Destination (Output)
1.  **`normalized_sessions.csv`**: The master dataset ready for Multiple Linear Regression (MLR) and Monte Carlo simulations. Contains luck-adjusted statistics and one-hot encoded gear vectors.

---

## 4. UI & Application Dependencies

The generated datasets sit at the core of the reporting ecosystem:

*   **`bbd_gui.py` (Real-Time UI):** Reads `gpph_sessions.csv` directly into memory on launch. It attempts to fuzzy-match live combat telemetry sessions (from `bbd_data/*.json`) to the historical Net GP/hr ledgers based on timestamps to render the live "Average NGP/s" dashboard element.
*   **`bbd_lab.py` (Streamlit Dashboard):** Acts as the primary interactive analytics interface. It relies on the unified `normalized_sessions.csv` to render its advanced telemetry visualizations.
*   **`plot_scripts/*.py`:** An entire suite of modular charting scripts (e.g., `plot_income_waterfall.py`, `plot_rng_waterfall.py`, `plot_monte_carlo.py`) directly dependencies on `normalized_sessions.csv` and `gpph_enriched.csv`.

---
## Summary of Constraints for New Collection Methods
If writing a new RuneLite plugin to replace or augment GP-Per-Hour:
1.  It **MUST** output the exact same 9 columns to `gpph_sessions.csv`.
2.  It **MUST** output the exact same 4 columns to `gpph_ledger_raw.csv`.
3.  Any timestamps used for financial linking must maintain the UTC 5-minute bucketing strategy to align with `enrich_gpph.py` historical lookups.
